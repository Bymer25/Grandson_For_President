
# GitFlow для GrandsonSound

В проекте используется стратегия ветвления, основанная на GitFlow с упрощениями для небольшой команды.

## Основные ветки
- main — стабильная версия, готовая к деплою. Все коммиты в main только через Pull Request из develop или hotfix.
- develop — основная ветка разработки. Содержит последние изменения, прошедшие ревью.

## Вспомогательные ветки
- feature/* — для новой функциональности. Создаётся от develop, после завершения вливается обратно через Pull Request.
  Пример: feature/add-payment, feature/player-controls.
- bugfix/* — для исправления ошибок в процессе разработки. Создаётся от develop, вливается через PR.
- hotfix/* — для срочных исправлений в продакшене. Создаётся от main, после исправления вливается и в main, и в develop.

## Правила именования веток
- Используйте нижний регистр и дефисы.
- Описывайте суть изменений кратко: feature/user-profile, bugfix/player-crash.

## Процесс работы
1. Создать задачу в Issues (опционально).
2. От develop создать ветку feature/название.
3. Выполнить изменения, сделать несколько коммитов.
4. Запушить ветку в репозиторий.
5. Открыть Pull Request в develop, запросить ревью.
6. После одобрения выполнить слияние (Merge) с удалением ветки.
7. Периодически (например, раз в спринт) создавать PR из develop в main для релиза.

## Коммиты
- Пишите понятные сообщения на русском или английском.
- Используйте префиксы: feat:, fix:, docs:, style:, refactor:, test:.
  Пример: feat: добавить страницу тарифов, fix: исправить переход по ссылкам.

## Тегирование версий
- При каждом релизе в main создаётся тег с номером версии (например, v1.0.0).
- Формат: v<major>.<minor>.<patch> (семантическое версионирование).
