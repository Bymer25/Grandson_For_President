# Стратегия PR strategy для GrandsonSound

## Цель
Обеспечить качество кода, единообразие и возможность обсуждения изменений перед попаданием в основную ветку.

## Когда создавать PR
- Для любой новой функциональности (ветка feature/*).
- Для исправления ошибок (ветка bugfix/*).
- Для срочных исправлений в продакшене (ветка hotfix/*).

## Требования к PR
1. Название: кратко описывает суть изменений. Желательно с префиксом (Feat, Fix, Docs).
   Пример: Feat: добавить страницу тарифов, Fix: исправить навигацию на мобильных.
2. Описание: что сделано, почему, как тестировалось. Если PR решает конкретную Issue, укажите Closes #номер.
3. Связанные задачи: ссылки на Issues (если есть).
4. Чеклист:
   - [ ] Код соответствует стилю проекта.
   - [ ] Изменения протестированы локально.
   - [ ] Нет конфликтов с целевой веткой.
   - [ ] Добавлены/обновлены комментарии (если необходимо).

## Процесс ревью
1. Автор назначает ревьюверов (минимум 1).
2. Ревьюверы проверяют код, оставляют комментарии.
3. Автор вносит правки, отвечает на комментарии.
4. После одобрения всех ревьюверов PR может быть смерджен.

## Критерии для ревью
- Функциональность: код делает то, что заявлено.
- Читаемость: понятные имена переменных, функции не слишком длинные.
- Архитектура: изменения не ломают существующую логику, соблюдаются принципы разделения ответственности.
- Безопасность: нет уязвимостей (XSS, инъекции и т.д.).
- Производительность: нет очевидных узких мест (например, запросы в цикле).

## Слияние (Merge)
- Используется стратегия Squash and merge для сохранения чистой истории в develop (один коммит на фичу).
- Ветка после слияния удаляется (если не требуется дальнейшая работа).

## Для hotfix-веток
- PR создаётся сразу в main.
- После слияния в main необходимо создать обратный PR в develop (или влить вручную), чтобы исправление попало и в основную ветку разработки.
